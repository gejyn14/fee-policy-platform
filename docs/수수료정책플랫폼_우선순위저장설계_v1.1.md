# 수수료 정책 플랫폼 — 정책 우선순위 저장·산정 설계서 v1.1

> 버전: v1.1 (2026-07-06) · 상위 진본: 기술설계서 v1.0 · 상태: 확정
> 대상 독자: 플랫폼 개발자, 원장 연계 담당, 아키텍처 리뷰어
> 범위: "계좌 무관 정책 우선순위"를 **어디에 저장하고 언제 재산정하는가**에 대한 결정과 근거. 프로토타입 구현과 프로덕션 매핑을 함께 규정한다.

---

## 1. 배경 — 우리가 답하려는 질문

체결 시 적용 수수료는 여러 정책(기본 BASE + 전 이벤트 EVENT)의 우선순위 경쟁에서 이긴 승자로 결정된다. 이 **우선순위 계산값을 어디에 저장하느냐**가 이 문서의 주제다.

두 가지를 반드시 구분한다.

| 개념 | 무엇인가 | 저장 성격 |
|---|---|---|
| **정책 우선순위 인덱스** | 계좌·체결가와 무관한, 정렬된 정책 순위(`rank`) | **파생값** — rules·schedules에서 언제든 재도출 |
| **계좌별 승자(바인딩)** | 특정 계좌+feeKey가 실제로 받는 요율표/룰 | **영속 대상** — `FEE_BINDING`(프로덕션) / `ResolveCache`(프로토타입) |

핵심 결론을 먼저 밝힌다:

> **정책 우선순위 인덱스는 전용 영속 테이블을 만들지 않는다.** 그것은 rules·schedules의 순수 파생값이므로 플랫폼 내부에서 계산하고, "저장된 우선순위"의 영속 실체는 계좌축 결과인 `FEE_BINDING`이 담당한다. 프로토타입에서는 인덱스를 스토어에 **메모이즈**하여 룰 변경 시에만 재산정한다.

---

## 2. 불변식 — 왜 "한 번 계산해 재사용"이 안전한가

정렬된 우선순위 인덱스는 순수하게 **`(rules, schedules, today)`의 함수**다. 이것이 성립하는 근거는 v0.8 통찰(같은 feeKey 안에서 승자는 가격에 따라 교차하지 않는다)이다:

- 해외파생 = 정액(계약당) → 낮은 정액이 항상 승(가격 무관).
- 주식·국내파생 = 정률(구간표 정액 add-on이 단조 보정) → 낮은 요율이 항상 승.
- 정률/정액이 같은 feeKey에 섞이지 않는다(상품군 구조가 일관).
- 우대(할인)는 기본을 전 구간에서 지배(≤)해야 성립 → 기본 vs 이벤트도 뒤집히지 않는다.
- 파생은 품목별 feeKey → 서로 겹치지 않는다.

따라서:

1. 같은 `(rules, schedules, today)`면 인덱스는 **불변**이다 → 안전하게 재사용 가능.
2. 계좌별 판정(대상 게이트 `isTarget`, 기간 게이트 `isBenefitActive`)은 인덱스가 아니라 **조회 시점**(`winnerFor`)의 O(N) 스캔에서 이뤄진다 → 인덱스 자체는 계좌와 무관하게 공유된다.
3. 협의(nego)는 인덱스에 편입되지 않고 `resolve`에서 별도 오버레이로 처리된다 → **nego 변경은 인덱스에 영향이 없다**.

이 세 성질이 "계좌 무관 인덱스를 한 번 만들어 모든 체결에 재사용" 설계를 정당화한다.

---

## 3. 저장 정책 — 전용 테이블을 만들지 않는 이유

계좌 무관 우선순위를 위한 전용 영속 테이블(예: `FEE_POLICY_PRIORITY`)을 **신설하지 않는다.** 근거 세 가지:

1. **이중 진실원(source of truth) 회피** — 인덱스는 `FEE_RULE`/`FEE_SCHEDULE`의 순수 파생값이다. 테이블로 굳히면 원본이 바뀔 때마다 동기화·정합성 검증 부담이 새로 생긴다.
2. **비싼 축은 계좌축** — 영속이 값어치 있는 건 O(계좌 × feeKey) 규모의 `FEE_BINDING`이다. 정책 순위는 O(룰 수)로 작아, 앱/잡 메모리에서 재빌드하는 편이 더 싸다.
3. **"원장은 룰을 모른다" 원칙** — 순위 산정은 플랫폼 내부 로직이며, 테이블 설계서 4.7절이 규정하듯 `FEE_BINDING`에는 **결과만** 남는다. 우선순위 로직을 원장이 알 필요가 없다.

**예외 — 그때 비로소 테이블이 정당화된다:**
- 플랫폼을 **다중 인스턴스로 수평 확장**하여 인스턴스별 로컬 캐시 정합성이 문제가 될 때 → 공유 캐시나 스냅샷 테이블.
- feeKey별 순위를 **시점별로 감사/재현**하거나 화면을 DB에서 직접 먹여야 할 때 → `FEE_POLICY_PRIORITY` 스냅샷.

v1.1 범위는 **단일 플랫폼 프로세스** 전제이므로 위 예외에 해당하지 않는다.

## 4. 아키텍처 — 2단 캐시

```
                 ┌──────────────────────────────────────────┐
   rules ────────►                                          │
   schedules ────► ① 정책 우선순위 인덱스 (계좌 무관, 파생)   │  ← 룰/요율표 변경 시에만 재산정
                 │    buildPolicyPriority(rules, sched, today)│
                 └───────────────┬──────────────────────────┘
                                 │ winnerFor(feeKey, acct, enr)  (O(N) 스캔 + 계좌 게이트)
                                 ▼
   account + feeKey ─► ② 계좌별 승자 (오버레이: 협의 grant 반영)
                                 │
                                 ▼
                 ┌──────────────────────────────────────────┐
                 │ 결과 저장:  ResolveCache(프로토) / FEE_BINDING(프로덕션)  │
                 └──────────────────────────────────────────┘
```

- **① 인덱스**는 계좌·체결가와 무관 → 룰이 바뀔 때만 갱신.
- **② 승자**는 계좌·협의를 얹은 최종 결과 → 계좌+feeKey 단위로 저장.

두 층을 분리했기에, 흔한(협의 없는) 계좌의 체결은 ①의 룩업 + ②의 얕은 스캔으로 끝난다.

## 5. 프로토타입 구현 (현재 레포)

이 레포에는 DB가 없다. zustand 스토어 + 인메모리 `Map`이 저장 계층이다. 인덱스는 스토어 모듈 스코프에 **메모이즈**한다.

### 5.1 메모 슬롯과 lazy 빌드

`src/store/useStore.ts`:

```ts
let priorityIdx: PolicyPriorityIndex | null = null;

// 계좌 무관 우선순위 인덱스 — (rules, schedules)의 파생. 없으면 빌드, 있으면 재사용.
function getPriorityIndex(s: Pick<State, 'rules' | 'schedules'>): PolicyPriorityIndex {
  return priorityIdx ??= buildPolicyPriority(s.rules, s.schedules, TODAY);
}
```

- `resolveFee`와 `policyPriority()`가 모두 `getPriorityIndex(s)`로 인덱스를 읽는다 → 캐시 미스 시에도 전체 정렬(O(N log N))을 반복하지 않고, 이미 정렬된 인덱스에서 `winnerFor`의 O(N) 스캔만 수행.

### 5.2 무효화 — 참조 변경 감지

무효화는 zustand `subscribe`로 `rules`/`schedules`의 **참조 변경**을 감지해 처리한다. 이 한 줄이 "룰/요율표를 바꾸는 mutation만 무효화, 그 외(nego·instrument)는 미영향"이라는 규칙을 DRY하게 실현한다.

```ts
// 룰/요율표가 바뀌면 우선순위 인덱스 무효화(다음 읽기에서 lazy 재빌드). nego·instrument 변경은 참조 유지 → 미영향.
useStore.subscribe((s, prev) => {
  if (s.rules !== prev.rules || s.schedules !== prev.schedules) priorityIdx = null;
});
```

무효화 동작 매트릭스(참조 변경 여부로 자동 결정됨):

| Mutation | rules/schedules 참조 변경 | 인덱스 무효화 |
|---|---|---|
| `approveRule` · `rejectRule` · `submitRule` · `extendNegotiated` · `addSchedule` · `batchActivateExpireRules` · `reset` | O (새 배열) | **O** |
| `submitNegoRequest` · `approveNegoRequest` · `rejectNegoRequest` · `applyNegoExtension` | X (`nego`만 변경) | **X** |
| `syncFromLedger` · `registerInstruments` · `batchRecomputeMetrics` | X (instruments/products만) | **X** |

> 모든 룰/요율표 mutation이 `.map()`/`filter()+spread`로 **새 배열**을 만들므로, 참조 동등 비교는 결코 under-invalidate 하지 않는다(간혹 no-op에 over-invalidate 하나 저비용). nego·instrument mutation은 `set()` partial에 `rules`/`schedules` 키를 담지 않아 zustand 얕은 병합이 참조를 보존한다 → 미영향.

### 5.3 계좌축 결과 캐시

계좌별 최종 승자는 기존 `ResolveCache`(`src/domain/cache.ts`, 인메모리 `Map`)가 `accountId + feeKey → {scheduleId, sourceRuleId, source}`로 저장한다. 이것이 프로덕션 `FEE_BINDING`의 프로토타입 대응물이다. 계좌·협의 변경 시 `invalidateAccount`/`invalidateByScope`로 무효화(기존 로직 유지).

## 6. 프로덕션 매핑

프로토타입의 "전역 메모이즈"는 프로덕션의 **rebind(바인딩) 잡 스코프 계산**을 단일 프로세스에서 흉내 낸 대역이다. 프로덕션에서 우선순위 인덱스는 오래 사는 캐시가 아니라 잡 안에서 잠깐 사는 transient 산출물이다.

```
[룰 변경/승인] → rebind 잡 실행
                 ├─ 우선순위 인덱스 계산 (잡 스코프, 잡 종료 시 소멸)
                 ├─ 계좌별 승자 확정 (협의 오버레이 포함)
                 └─ FEE_BINDING UPSERT          ← 영속 저장(계좌축 결과)
[체결] → 원장이 FEE_BINDING 룩업 (index-only access) → 끝
         (원장은 룰도 우선순위도 모른다)
```

| | 프로토타입(현재 레포) | 프로덕션 |
|---|---|---|
| 우선순위 인덱스 | 스토어 메모이즈(전역, lazy) | rebind 잡 스코프(transient) |
| 계좌축 결과 저장 | `ResolveCache`(인메모리 `Map`) | `FEE_BINDING` 테이블 |
| 무효화 트리거 | 룰 변경 + (배치가 대변하는) 날짜 | 룰 변경 이벤트 → rebind + 일배치(날짜 롤오버) |

**패턴("파생값, 룰 변경 때 재계산")은 프로덕션까지 그대로 유지되고, 저장 메커니즘만 "프로세스 로컬 `Map` → 잡 스코프 계산 + `FEE_BINDING` 영속"으로 바뀐다.** 체결(hot path)은 어느 환경에서도 인덱스를 건드리지 않는다.

## 7. 날짜 롤오버 caveat

`inRanking`이 `today`(기간 게이트)에 의존하므로, 이벤트의 시작/종료로 순위 편입이 날짜에 따라 바뀔 수 있다. 따라서:

- **프로덕션**: 무효화 트리거에 "룰 변경"뿐 아니라 **날짜 롤오버(일 1회)** 를 반드시 포함해야 한다.
- **프로토타입**: `TODAY`가 상수라 세션 중 저절로 stale되지 않는다. 날짜 경과는 `batchActivateExpireRules`가 룰 status를 플립하는 것으로 대변되고, 이는 `rules` 변경 → 자동 무효화로 이어지므로 별도 날짜 타이머가 불필요하다.

## 8. 검증(테스트)

`src/store/useStore.test.ts`의 `정책 우선순위 사전 산정` 블록:

1. **메모이즈**: 연속 `policyPriority()` 호출이 **동일 인스턴스**를 반환(참조 동등).
2. **무효화 정확성**: `approveRule`로 더 싼 룰을 활성화하면 다음 인덱스에서 승자가 갱신됨(`topFor`가 신규 룰을 최상위로).
3. **비무효화(회귀 방지)**: `approveNegoRequest` 후 인덱스 인스턴스가 유지됨(협의는 인덱스 무영향).
4. **등가성**: 메모된 인덱스의 `winnerFor` 승자 == 계좌 오버레이 없는 계좌의 `resolveFee` 승자(기존 테스트가 커버).

전체 스위트 146/146 통과, `tsc`·`lint` 클린으로 회귀 없음을 확인한다.
