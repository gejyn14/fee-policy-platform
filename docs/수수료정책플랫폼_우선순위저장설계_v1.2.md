# 수수료 정책 플랫폼 — 정책 우선순위 저장·산정 설계서 v1.2

> 버전: v1.2 (2026-07-07) · 상위 진본: 기술설계서 v1.0 · 전신: v1.1 (2026-07-06) · 상태: **리뷰 대기**
> 대상 독자: 플랫폼 개발자, 원장 연계 담당, 아키텍처 리뷰어
> 범위: 정책(기본·이벤트·협의) 우선순위를 **어떻게 하나의 룰 모델로 통일하고, 계좌×조회키 전체 판을 어디에 어떻게 사전 산출·저장하는가**. 표기 원칙: 본문은 국문, 테이블·칼럼·코드값 등 식별자는 **전부 영문**.

---

## 1. 배경 — v1.1에서 무엇이 달라지나

v1.1은 "계좌 무관 정책 우선순위 인덱스를 어디에 저장하는가"에 답했다: 인덱스는 rules·schedules의 파생값이므로 전용 테이블 없이 메모이즈하고, 영속 실체는 계좌축 결과(`FEE_BINDING`)가 담당한다.

v1.2는 질문을 확장한다. 현업 검토(2026-07-07)에서 확인된 세 가지 사실이 출발점이다:

1. **협의수수료는 정형화되어 있다.** 협의 요율은 홈페이지에 공시 가능한 수준의 표준 등급표다. 계좌별로 다른 것은 요율이 아니라 "어느 등급을 승인받았고 언제까지 유효한가"뿐이다. 승인·연장이라는 수명주기 차이는 있으나, 요율 구조 관점에서 기본·이벤트와 다르지 않다.
2. **원장 체결에는 조회에 필요한 정보가 전부 들어온다.** 자산군·거래소·상품분류(선물/옵션, 주식/ETF)·품목·채널. 따라서 판의 키를 합성 문자열이 아닌 **명시 칼럼**으로 전개할 수 있다.
3. **파생은 예외 없이 상품(품목)별로 수수료 정책이 짜인다.** 판의 입도는 이를 수용해야 한다.

이에 따라 v1.2의 핵심 결정:

> **① 협의(NEGOTIATED)를 기본(BASE)·이벤트(EVENT)와 동일한 룰 모델·동일한 요율 랭킹에 편입한다.**
> **② 계좌×조회키 전체 판을 일배치로 사전 산출하여 `FEE_BINDING` 한 테이블에서 요율표 ID를 관리한다** (수시 증분 갱신 병행).
> **③ 판의 키에 조회키(`LOOKUP_KEY`) 축을 신설한다** — 파생은 `FUTURES`/`OPTIONS`(품목별 행 유지), 해외주식은 `STOCK`/`ETF`.
> **④ 특정 계좌×조회키에 "어떤 정책이 적용되었는가"를 `SOURCE_RULE_ID`·`SOURCE_TYPE`(현재)과 `FEE_BINDING_HISTORY`(이력)로 관리한다.**

v1.1의 결론 중 유지되는 것: **룰축 우선순위 인덱스는 여전히 전용 영속 테이블을 만들지 않는다**(배치 잡 스코프 transient / 프로토타입 메모이즈). 바뀌는 것은 계좌축 판의 산출 방식(룰 변경 시 rebind → **일배치 전체 재산출 + 수시 증분**)과 커버리지(협의 포함 전체 판)다.

## 2. 정책 모델 일원화 — 협의도 하나의 룰

### 2.1 구조

세 정책 타입이 동일한 모델을 공유한다: **룰(`FEE_RULE`) + 요율표(`FEE_SCHEDULE`) + 적용 게이트**.

| RULE_TYPE | 룰이 표현하는 것 | 적용 게이트(계좌별 판정) | 수명주기 |
|---|---|---|---|
| `BASE` | 자산군·범위별 기본 요율 | 없음(항상 적용 후보) | 상시 |
| `EVENT` | 기간 한정 우대 요율 | 대상 게이트(신청/가입/휴면복귀/일괄) + 기간 게이트 | 시작~종료 |
| `NEGOTIATED` | **공시 가능한 표준 협의 등급** (예: 해외파생 협의 TIER2 = $0.80/계약) | **유효기간 내 활성 enrollment 보유** | 승인 → 연장 반복 |

계좌별 차이는 룰이 아니라 **적용 관계(`FEE_ENROLLMENT`)**로 표현한다:

- **승인** = 계좌×협의룰 enrollment 생성(`STATUS='ACTIVE'`, `VALID_FROM`/`VALID_TO` 부여).
- **연장** = 해당 enrollment의 `VALID_TO` 갱신. 조건 평가(`FEE_CONDITION_EVAL_LOG`, 6개월 평균자산/약정액) 기반의 자동연장/승인후연장 프로세스는 기존 그대로 유지된다 — 바뀌는 것은 "무엇이 연장되는가"(협의 예외 레코드 → enrollment)뿐이다.
- **영업 예외(bypass) 승인**도 enrollment의 속성(`QUALIFY_TYPE='EXCEPTION'`, `REASON`)으로 수용한다.

이 일원화로 v1.1의 "협의는 순위 밖 오버레이" 설계는 폐기된다. 협의 요율이 비표준 개별 협상이라는 전제가 사실이 아니기 때문이다.

### 2.2 FEE_ENROLLMENT 확장 (DDL)

```sql
ALTER TABLE FEE_ENROLLMENT ADD (
    VALID_FROM    DATE,                                        -- NEGOTIATED: 승인일. EVENT 신청형: 가입일
    VALID_TO      DATE,                                        -- NEGOTIATED: 연장 시 UPDATE
    STATUS        VARCHAR2(10) DEFAULT 'ACTIVE' NOT NULL,      -- REQUESTED / ACTIVE / REJECTED / EXPIRED
    QUALIFY_TYPE  VARCHAR2(10),                                -- MET(조건충족) / EXCEPTION(영업 bypass)
    REASON        VARCHAR2(200),                               -- bypass·반려 사유
    APPROVED_BY   VARCHAR2(50),
    APPROVED_AT   TIMESTAMP
);
ALTER TABLE FEE_ENROLLMENT ADD CONSTRAINT CK_FEE_ENROLLMENT_STATUS
    CHECK (STATUS IN ('REQUESTED','ACTIVE','REJECTED','EXPIRED'));
ALTER TABLE FEE_ENROLLMENT ADD CONSTRAINT CK_FEE_ENROLLMENT_QUALIFY
    CHECK (QUALIFY_TYPE IS NULL OR QUALIFY_TYPE IN ('MET','EXCEPTION'));
```

## 3. 통합 랭킹 — 세 타입이 한 순위에서 경쟁

### 3.1 정렬 기준

우선순위 인덱스는 `(rules, schedules, today)`의 순수 파생값이라는 v1.1 불변식을 유지하되, 편입 대상을 `BASE`+`EVENT`에서 **`BASE`+`EVENT`+`NEGOTIATED`**로 확장한다. 정렬:

1. **`rank` 오름차순** — 구조 그룹 순위값(정률=요율 합, 정액=정액 합, 구간표=대표 구간 요율+최소). 최저 요율이 이긴다.
2. 동률 시 **`NEGOTIATED` > `EVENT` > `BASE`** (고객 약정 > 판촉 > 기본).
3. 동률 시 **적용범위 구체성**(제약 차원 수) 높은 쪽.
4. 동률 시 **RULE_ID** (결정성).

### 3.2 자격 게이트 (조회·산출 시점의 계좌별 판정)

| SOURCE_TYPE | 게이트 |
|---|---|
| BASE | 통과(항상) |
| EVENT | 대상 게이트(`isTarget`: 신청/가입/휴면복귀/일괄 대상) AND 기간 게이트(`isBenefitActive`: 절대형=룰 기간, 상대형=계좌별 가입일 기준 윈도) |
| NEGOTIATED | `FEE_ENROLLMENT`에 `STATUS='ACTIVE'`이고 `VALID_FROM <= 기준일 <= VALID_TO`인 행 존재 |

**의미 변화(v1.1 대비):** 협의가 무조건 우선이 아니다. 이벤트가 협의보다 싸면 이벤트가 이긴다(고객 유리 원칙). 협의 약정을 계약상 반드시 적용해야 하는 케이스가 발견되면 tie-break이 아니라 별도 계층으로 승격해야 하며, 이는 v1.2에서 채택하지 않은 대안으로 기록해둔다.

### 3.3 예시 — A-1001의 해외 ETF 셀에서의 경쟁

| 후보 | rank(요율) | 게이트 | 결과 |
|---|---|---|---|
| R-EVT-12 (0.09%) | 9.0 | 신청형 — enrollment 9002 ACTIVE → 통과 | **승자** |
| R-BASE-01 (0.25%) | 25.0 | 통과 | 2위 |

해외파생 ES선물 셀(A-1001이 TIER2 협의 보유):

| 후보 | rank | 게이트 | 결과 |
|---|---|---|---|
| R-NEGO-03 ($0.50) | 0.50 | enrollment 없음 → 탈락 | — |
| R-NEGO-02 ($0.80) | 0.80 | enrollment 9001 ACTIVE → 통과 | **승자** |
| R-BASE-05 ($2.50) | 2.50 | 통과 | 2위 |

## 4. 조회키(LOOKUP_KEY) — 판의 "무엇을 거래하나" 축

### 4.1 정의

조회키는 원장이 체결 시 수수료를 조회하는 상품분류 축이다. 자산군별로 입도가 다르다:

| ASSET_CLASS | LOOKUP_KEY | PRODUCT_CODE | 근거 |
|---|---|---|---|
| `DOMESTIC_DERIV` / `OVERSEAS_DERIV` | `FUTURES` / `OPTIONS` | **품목(기초자산)코드** — 예: ES, GC, KOSPI200 | 파생은 예외 없이 품목별로 정책이 짜인다 → 품목별 행 유지, 조회키는 그 상위 분류 |
| `OVERSEAS_STOCK` | `STOCK` / `ETF` | `'*'` | 주식형은 종목 차원 없음(v0.5 불변식). ETF는 주식과 요율이 달라 조회키로 분리 |
| `DOMESTIC_STOCK` | `STOCK` (필요 시 `ETF` 확장) | `'*'` | |
| `GOLD_SPOT` | `GOLD` | `'*'` | |

### 4.2 키 차원 배치 원칙

> **그 차원에 따라 이기는 룰(요율표)이 달라지면 판의 키, 같은 요율표 안에서 값만 다르면 요율표 내부 차원.**

| 차원 | 배치 | 이유 |
|---|---|---|
| ASSET_CLASS · LOOKUP_KEY · PRODUCT_CODE(파생) | **판의 키** | 분류별로 승자 자체가 다름 |
| EXCHANGE_CODE | **판의 키** (`'*'` 허용) | 해외주식 시장별(미국/일본/홍콩 등) 요율 차등이 승자를 바꿈 |
| CHANNEL_CODE | **판의 키** (`'*'` 허용) | 매체 한정 이벤트("MTS만 할인")는 매체에 따라 승자가 바뀜 — v1.0 4.6절 결정 유지 |
| 세션(주간/야간) | 요율표 내부(`FEE_COMPONENT.SESSION_CODE`) | 승자는 같고 요율만 다름 — v1.0 4.5절 결정 유지 |
| 매수/매도 | 요율표 내부(필요 시 `FEE_COMPONENT`에 SIDE 차원 추가) | 통상 동일 요율. 차등 필요 시에도 승자는 불변 |

### 4.3 판의 행 수 추정

계좌당 행 수 = 파생 활성품목 수 × 2(선물/옵션) + 해외주식 2(주식/ETF) × 시장 수 + 국내주식 1 + 금현물 1 수준으로 유계(bounded)다. 계좌가 개설한 상품군의 셀만 생성하면(계좌-상품군 가입 정보 기준 pruning) 100만 계좌 기준 수천만 행 규모로, 커버링 인덱스 단일 테이블로 충분히 관리 가능하다.

## 5. 판 테이블 — FEE_BINDING v1.2 (DDL·예시·조회)

기존 `SCOPE_KEY VARCHAR2(50)`(`거래소:품목` 합성 문자열)를 명시 칼럼으로 분해하고, `LOOKUP_KEY`·`SOURCE_TYPE`을 신설한다. 코드값도 전부 영문으로 통일한다.

### 5.1 DDL

```sql
CREATE TABLE FEE_BINDING (
    ACCOUNT_ID      VARCHAR2(20)   NOT NULL,
    ASSET_CLASS     VARCHAR2(20)   NOT NULL,   -- DOMESTIC_STOCK / OVERSEAS_STOCK / DOMESTIC_DERIV / OVERSEAS_DERIV / GOLD_SPOT
    EXCHANGE_CODE   VARCHAR2(10)   DEFAULT '*' NOT NULL,   -- '*' = 전 거래소 공통
    LOOKUP_KEY      VARCHAR2(10)   NOT NULL,   -- FUTURES / OPTIONS / STOCK / ETF / GOLD
    PRODUCT_CODE    VARCHAR2(20)   DEFAULT '*' NOT NULL,   -- 파생만 품목코드, 그 외 '*'
    CHANNEL_CODE    VARCHAR2(10)   DEFAULT '*' NOT NULL,   -- MTS / HTS / BRANCH / '*'
    VALID_FROM      DATE           NOT NULL,
    VALID_TO        DATE           NOT NULL,
    SCHEDULE_ID     VARCHAR2(30)   NOT NULL,   -- 적용 요율표 (원장이 읽는 값)
    SOURCE_RULE_ID  VARCHAR2(30)   NOT NULL,   -- 적용 정책(룰). 협의도 룰 ID로 추적
    SOURCE_TYPE     VARCHAR2(10)   NOT NULL,   -- BASE / EVENT / NEGOTIATED
    REASON          VARCHAR2(200),             -- 예: 'NEGOTIATED TIER2 won by lowest rate'
    UPDATED_AT      TIMESTAMP      DEFAULT SYSTIMESTAMP NOT NULL,
    CONSTRAINT FK_FEE_BINDING_SCHEDULE FOREIGN KEY (SCHEDULE_ID) REFERENCES FEE_SCHEDULE (SCHEDULE_ID),
    CONSTRAINT FK_FEE_BINDING_RULE     FOREIGN KEY (SOURCE_RULE_ID) REFERENCES FEE_RULE (RULE_ID),
    CONSTRAINT CK_FEE_BINDING_ASSET  CHECK (ASSET_CLASS IN ('DOMESTIC_STOCK','OVERSEAS_STOCK','DOMESTIC_DERIV','OVERSEAS_DERIV','GOLD_SPOT')),
    CONSTRAINT CK_FEE_BINDING_LOOKUP CHECK (LOOKUP_KEY IN ('FUTURES','OPTIONS','STOCK','ETF','GOLD')),
    CONSTRAINT CK_FEE_BINDING_SOURCE CHECK (SOURCE_TYPE IN ('BASE','EVENT','NEGOTIATED'))
);

-- PK 제약과 커버링 인덱스 겸용: 체결 조회가 인덱스 블록만으로 끝나도록(index-only access)
-- 결과 칼럼(SCHEDULE_ID, SOURCE_RULE_ID, SOURCE_TYPE)까지 포함해 생성한다.
CREATE UNIQUE INDEX PK_FEE_BINDING ON FEE_BINDING
    (ACCOUNT_ID, ASSET_CLASS, EXCHANGE_CODE, LOOKUP_KEY, PRODUCT_CODE, CHANNEL_CODE,
     VALID_FROM, VALID_TO, SCHEDULE_ID, SOURCE_RULE_ID, SOURCE_TYPE);

ALTER TABLE FEE_BINDING ADD CONSTRAINT PK_FEE_BINDING
    PRIMARY KEY (ACCOUNT_ID, ASSET_CLASS, EXCHANGE_CODE, LOOKUP_KEY, PRODUCT_CODE, CHANNEL_CODE, VALID_FROM)
    USING INDEX PK_FEE_BINDING;
```

PK에 `VALID_FROM`을 포함하는 이유(v1.0과 동일): 미래 시작 바인딩(예: 다음 주 시작 이벤트를 오늘 승인)이 현재 유효 구간과 공존한다. 구간 비중첩은 배치·증분 산출기(쓰기 주체)가 애플리케이션 레벨에서 보장하고, 일배치 diff가 위반을 탐지한다.

### 5.2 예시 — A-1001의 판 (2026-07-07 기준)

§2.3·§3.3 시나리오가 판에 놓인 모습:

| ACCOUNT_ID | ASSET_CLASS | EXCHANGE_CODE | LOOKUP_KEY | PRODUCT_CODE | CHANNEL_CODE | VALID_FROM | VALID_TO | SCHEDULE_ID | SOURCE_RULE_ID | SOURCE_TYPE |
|---|---|---|---|---|---|---|---|---|---|---|
| A-1001 | OVERSEAS_DERIV | CME | FUTURES | ES | * | 2026-01-02 | 2026-12-31 | SCH-OD-NEGO-T2 | R-NEGO-02 | NEGOTIATED |
| A-1001 | OVERSEAS_DERIV | CME | FUTURES | GC | * | 2026-01-02 | 2026-12-31 | SCH-OD-NEGO-T2 | R-NEGO-02 | NEGOTIATED |
| A-1001 | OVERSEAS_DERIV | CME | OPTIONS | ES | * | 2026-01-01 | 9999-12-31 | SCH-OD-BASE | R-BASE-05 | BASE |
| A-1001 | OVERSEAS_STOCK | * | ETF | * | * | 2026-07-01 | 2026-12-31 | SCH-OS-EVT-03 | R-EVT-12 | EVENT |
| A-1001 | OVERSEAS_STOCK | * | STOCK | * | * | 2026-01-01 | 9999-12-31 | SCH-OS-BASE | R-BASE-01 | BASE |

읽는 법:

- ES·GC **선물**은 협의 TIER2($0.80)가 승자 — 협의 유효기간(enrollment의 VALID_FROM/TO)이 행의 유효기간으로 전파된다.
- ES **옵션**은 협의 범위 밖이라 기본($2.50). 같은 품목이라도 조회키(FUTURES/OPTIONS)가 다르면 다른 행 — 조회키 축 신설의 효과.
- 해외 **ETF**는 이벤트(0.09%)가, **주식**은 기본(0.25%)이 승자 — 주식형에서 조회키가 STOCK/ETF를 분리한 효과.
- 상대형(가입일 기준) 이벤트는 계좌별 혜택 윈도가 `VALID_FROM`/`VALID_TO`에 그대로 반영된다(판이 계좌 단위이므로 표현 가능).

### 5.3 원장 체결 조회 (계약)

```sql
SELECT SCHEDULE_ID, SOURCE_RULE_ID, SOURCE_TYPE
FROM   FEE_BINDING
WHERE  ACCOUNT_ID    = :account_id
  AND  ASSET_CLASS   = :asset_class
  AND  LOOKUP_KEY    = :lookup_key                    -- 파생: FUTURES/OPTIONS, 해외주식: STOCK/ETF
  AND  EXCHANGE_CODE IN (:exchange_code, '*')
  AND  PRODUCT_CODE  IN (:product_code, '*')
  AND  CHANNEL_CODE  IN (:channel_code, '*')
  AND  VALID_FROM <= :trade_date AND :trade_date <= VALID_TO
ORDER  BY CASE WHEN PRODUCT_CODE  = :product_code  THEN 0 ELSE 1 END,
          CASE WHEN CHANNEL_CODE  = :channel_code  THEN 0 ELSE 1 END,
          CASE WHEN EXCHANGE_CODE = :exchange_code THEN 0 ELSE 1 END
FETCH  FIRST 1 ROW ONLY;
```

- 구체 행(품목·채널·거래소 한정)이 `'*'` 행보다 우선. 체결에 들어오는 정보(자산군·거래소·상품분류·품목·채널)만으로 조회가 완결된다.
- 0건이면 기존 등급 요율 fallback(v1.0 4.3절 유지). 원장 수정 범위는 이 분기 한 곳뿐이라는 계약도 유지된다.
- **원장은 여전히 룰도 우선순위도 모른다** — 랭킹·게이트는 전부 산출기(배치/증분) 안에서 끝나고, 판에는 결과만 남는다.

## 6. 산출 — 일배치 전체 재산출 + 수시 증분

### 6.1 일배치 `BATCH_REBUILD_FEE_BINDING` (매일 1회, 장 마감 후)

```
① 활성 룰(BASE+EVENT+NEGOTIATED)·요율표 로드
   → 우선순위 인덱스 빌드 (잡 스코프 transient — v1.1 원칙 유지, 전용 테이블 없음)
② 셀 유니버스 전개: 계좌 × (ASSET_CLASS, EXCHANGE, LOOKUP_KEY, PRODUCT[파생만])
   — 계좌가 개설한 상품군의 셀만 생성. 채널 축은 매체 한정 룰이 승자인 셀에만 추가 행
③ 셀별 승자 확정: 랭킹을 내려가며 자격 게이트(§3.2) 통과 첫 정책
④ 스테이징 vs 현행 FEE_BINDING diff
   → UPSERT + FEE_BINDING_HISTORY 적재(TRIGGER_SOURCE='DAILY_REBUILD')
   → diff 리포트 = reconciliation 산출물 (v1.0의 별도 reconciliation 배치를 흡수)
```

전체 재산출이 매일 돌므로 **날짜 롤오버 무효화(v1.1 §7 caveat)가 구조적으로 해소**된다: 이벤트 시작/종료·협의 만료가 다음 배치에서 반드시 반영되고, 판은 매일 "룰 전체로부터 재유도한 기대값"과 일치함이 보장된다.

### 6.2 수시 증분 (주중 이벤트 발생 시, 당일 적용 보장)

| 트리거 | 영향 범위 | TRIGGER_SOURCE |
|---|---|---|
| 룰 승인/반려/조기종료 | 룰 scope에 매칭되는 셀 × 대상 계좌 | RULE_APPROVED / RULE_EXPIRED |
| 협의 승인 | 해당 계좌 × 협의룰 scope 셀 | NEGO_APPROVED |
| 협의 연장(자동/승인후) | 해당 계좌 × 협의룰 scope 셀 (VALID_TO 연장) | NEGO_EXTENDED |
| 이벤트 신청/가입 | 해당 계좌 × 이벤트 scope 셀 | ENROLLMENT |
| 휴면복귀 감지 | 해당 계좌 × 휴면복귀형 룰 scope 셀 | DORMANT_RETURN |

증분은 일배치와 **동일한 승자 확정 로직**을 좁은 범위에 적용하는 것이므로 로직이 두 벌 존재하지 않는다(단일 산출 함수 공유). 증분이 실패·누락되어도 다음 일배치가 보정한다 — 증분은 latency 최적화, 일배치가 정합성의 진본.

### 6.3 무효화 매트릭스 (v1.1 §5.2 개정)

| 변경 | 룰축 우선순위 인덱스 | 계좌축 판(FEE_BINDING) |
|---|---|---|
| 룰 기안/승인/반려/종료 (표준 협의 등급 개정 포함) | **무효화**(재빌드) | 증분 갱신 + 일배치 |
| 협의 승인/연장/반려 (enrollment 변경) | 영향 없음(룰 불변) | **해당 계좌 증분 갱신** |
| 이벤트 신청/가입 | 영향 없음 | 해당 계좌 증분 갱신 |
| 계좌·종목 마스터 동기화 | 영향 없음 | 일배치(셀 유니버스 갱신) |
| 날짜 경과 | 일배치가 매일 재빌드 | 일배치가 매일 재산출 |

v1.1과의 차이: 협의 승인/연장이 "인덱스·판 모두 무영향"에서 "인덱스 무영향 + **판 증분 갱신**"으로 바뀐다. 협의가 판에 편입되었기 때문이다.

## 7. 적용 정책 관리 — "이 계좌 이 조회키에 무엇이 적용되었나"

### 7.1 현재 적용 정책 = 판 그 자체

`SOURCE_RULE_ID` + `SOURCE_TYPE`이 답이다. 협의가 룰로 일원화된 덕에 협의 적용도 룰 ID로 추적된다(v1.1에서는 협의 승자의 `sourceRuleId`가 null이었다).

```sql
-- A-1001의 해외 ETF에 지금 적용 중인 정책
SELECT SCHEDULE_ID, SOURCE_RULE_ID, SOURCE_TYPE, VALID_FROM, VALID_TO, REASON
FROM   FEE_BINDING
WHERE  ACCOUNT_ID = 'A-1001' AND ASSET_CLASS = 'OVERSEAS_STOCK' AND LOOKUP_KEY = 'ETF'
  AND  VALID_FROM <= :today AND :today <= VALID_TO;
-- → SCH-OS-EVT-03 / R-EVT-12 / EVENT / 2026-07-01 ~ 2026-12-31
```

### 7.2 변경 이력 = FEE_BINDING_HISTORY

키 칼럼을 판과 동일하게 확장하고 old/new 값과 트리거를 남긴다:

```sql
CREATE TABLE FEE_BINDING_HISTORY (
    HISTORY_ID          NUMBER(19)    NOT NULL,
    ACCOUNT_ID          VARCHAR2(20)  NOT NULL,
    ASSET_CLASS         VARCHAR2(20)  NOT NULL,
    EXCHANGE_CODE       VARCHAR2(10)  DEFAULT '*' NOT NULL,
    LOOKUP_KEY          VARCHAR2(10)  NOT NULL,
    PRODUCT_CODE        VARCHAR2(20)  DEFAULT '*' NOT NULL,
    CHANNEL_CODE        VARCHAR2(10)  DEFAULT '*' NOT NULL,
    OLD_SCHEDULE_ID     VARCHAR2(30),
    OLD_SOURCE_RULE_ID  VARCHAR2(30),
    OLD_SOURCE_TYPE     VARCHAR2(10),
    NEW_SCHEDULE_ID     VARCHAR2(30),
    NEW_SOURCE_RULE_ID  VARCHAR2(30),
    NEW_SOURCE_TYPE     VARCHAR2(10),
    TRIGGER_SOURCE      VARCHAR2(20),   -- DAILY_REBUILD / RULE_APPROVED / NEGO_APPROVED / NEGO_EXTENDED / ENROLLMENT / DORMANT_RETURN / RULE_EXPIRED
    CHANGE_REASON       VARCHAR2(200),
    CHANGED_AT          TIMESTAMP     DEFAULT SYSTIMESTAMP NOT NULL,
    CONSTRAINT PK_FEE_BINDING_HISTORY PRIMARY KEY (HISTORY_ID)
);
CREATE INDEX IDX_FEE_BINDING_HIST ON FEE_BINDING_HISTORY
    (ACCOUNT_ID, ASSET_CLASS, LOOKUP_KEY, CHANGED_AT);
```

### 7.3 예시 — "왜 7월부터 ETF 수수료가 내려갔나" (민원 대응)

| CHANGED_AT | ACCOUNT_ID | ASSET_CLASS | LOOKUP_KEY | OLD → NEW SCHEDULE | OLD → NEW SOURCE | TRIGGER_SOURCE |
|---|---|---|---|---|---|---|
| 2026-07-01 05:10 | A-1001 | OVERSEAS_STOCK | ETF | SCH-OS-BASE → SCH-OS-EVT-03 | R-BASE-01(BASE) → R-EVT-12(EVENT) | DAILY_REBUILD |
| 2026-01-02 14:22 | A-1001 | OVERSEAS_DERIV | FUTURES | SCH-OD-BASE → SCH-OD-NEGO-T2 | R-BASE-05(BASE) → R-NEGO-02(NEGOTIATED) | NEGO_APPROVED |

(두 번째 건은 협의 scope 내 품목(ES·GC)별로 각 1행씩 적재된 것을 대표 1행으로 표기)

체결 명세에는 `SCHEDULE_ID`·`SOURCE_RULE_ID`가 스탬프되므로(원장 체결 로그), "과거 특정 체결에 적용된 정책"은 체결 로그에서, "적용 정책의 변천"은 HISTORY에서, "현재"는 판에서 — 3계층으로 완결된다.

## 8. 프로토타입 매핑 (현재 레포 → v1.2 목표 모델)

현 프로토타입(v0.8)은 v1.1 모델(협의 오버레이)로 구현되어 있다. v1.2 반영 시 변경 항목:

| 항목 | 현재 (v1.1 모델) | v1.2 목표 |
|---|---|---|
| 협의 표현 | `NegoException`(계좌별 예외 레코드, `resolve` 오버레이) | `FeeRule(type: 'NEGOTIATED')` 표준 등급 + `Enrollment(validFrom/validTo)` |
| 랭킹 편입 | BASE+EVENT (`buildPolicyPriority`) | BASE+EVENT+NEGOTIATED, tie-break에 SOURCE_TYPE 순서 추가 |
| `resolve` | 협의 우선 분기 → 없으면 `winnerFor` | 협의 분기 삭제, `winnerFor` 단일 경로 (NEGOTIATED 게이트 = enrollment 유효성) |
| feeKey | `자산군\|거래소\|세션\|채널(\|품목)` | 세션 제거·`lookupKey` 추가: `자산군\|거래소\|조회키\|채널(\|품목)` |
| 판 저장 | `ResolveCache`(인메모리 Map, lazy) | `batchRebuildBindings`(전체 판 사전 산출) + 증분 갱신, UI에 판·이력 화면 |
| 인덱스 메모이즈 | 스토어 메모이즈 + 참조변경 무효화 | 유지 (무효화 매트릭스만 §6.3로 개정) |

구현은 본 문서 범위 밖(후속 구현 계획서에서 태스크화).

## 9. v1.1 대비 변경 요약

| 항목 | v1.1 | v1.2 (본 문서) |
