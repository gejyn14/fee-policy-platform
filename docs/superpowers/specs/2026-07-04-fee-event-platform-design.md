# 수수료 이벤트 정형화 플랫폼 — 설계 문서

- 작성일: 2026-07-04
- 상태: 사용자 승인 완료 (섹션별 승인)
- 다음 단계: 구현 계획 수립 → 프로토타입 웹앱 구현

## 1. 배경과 문제

현재 상태:

- 계좌에 수수료 등급이 있고, 품목(상품·종목)별로 등급별 수수료가 리스트업되어 있다.
- 체결이 일어나면 수수료를 거래내역과 잔고에 near-instant로 반영한다.
- 수수료 이벤트에 정형화된 구조가 없어 **이벤트마다 개발 건**으로 처리된다.
- 협의수수료는 담당자가 수수료 수익·재산현황을 엑셀로 sum-up하여 수작업으로 기간을 연장한다.

목표:

1. 수수료 이벤트를 원장에 **데이터로 등록**하면 개발 없이 적용되는 정형화 구조.
2. 기본수수료(등급)·이벤트·협의수수료를 **하나의 룰 모델**로 통합하고, 시스템이 자동으로 **고객에게 가장 유리한 수수료를 적용**.
3. 협의수수료 조건 평가(6개월 롤링 지표) 자동화로 엑셀 수작업 대체.
4. 현업 담당자가 bulk 업로드 등으로 **손쉽게 이벤트를 등록·운영**할 수 있는 관리 웹.

적용 상품군: 국내주식, 해외주식, 국내파생, 해외파생, 금현물 (전 상품군).

## 2. 핵심 설계 결정

| 결정 | 내용 | 근거 |
|---|---|---|
| 사전 바인딩 방식 | 체결 시점 룰 평가가 아니라, 변경 발생 시점에 "적용될 요율표"를 미리 계좌×범위 단위로 바인딩. 원장은 조회만 한다. | 체결 hot path 단순화(near-instant 보장), 새 이벤트 유형이 생겨도 원장 수정 0건, 적용 근거의 물질화(감사·민원 대응) |
| 요율표 바인딩 (숫자 아님) | 바인딩 대상은 스칼라 요율이 아니라 요율표 참조. 구간 판정(국내옵션 구간별 수수료 등)만 체결 시점에 원장이 수행. | 구간별·세션별·통화별 차등은 사전 확정 불가 |
| 변경은 즉시, 배치는 검증 | 신청·연장·등급변경 등은 이벤트 드리븐으로 즉시 바인딩 갱신. 일 배치는 재계산이 아니라 정합성 검증(reconciliation) 역할. | 배치 단독 의존 시 트리거 누락 = 잘못된 수수료. 안전망 필요 |
| 기본수수료도 룰 | 기존 등급 체계를 BASE 타입 룰로 이관하여 기본/이벤트/협수가 같은 모델에서 경쟁. | "여러 수수료 중 최적 선택" 요구의 전제 |
| 최저가 자동 적용 | 적용 가능한 룰 중 고객부과 총액이 가장 낮은 것을 시스템이 자동 선택. 동률 tie-break: 협수 > 이벤트 > 기본. | 고객 액션 불필요, 민원 소지 최소화 |
| 등록 시점 지배관계 검증 | 이벤트 요율표는 기존 적용 요율 대비 **전 구간에서 같거나 싸야** 승인 가능. | 구간표끼리 교차(A가 저가 구간에서, B가 고가 구간에서 유리)하면 최저가 판정 불가. 검증을 강제하면 바인딩이 항상 단일로 확정되고 원장에 min() 로직이 불필요 |
| 유관기관분 회사 부담 지원 | 요율 구성요소마다 부담주체(고객부과/회사부담/면제)를 지정. "완전 무료" 이벤트에서 유관기관 수수료를 회사가 대납 가능. | 실제 이벤트 요구. 역마진 가드로 보완 |
| 원장 DB 공유 | 바인딩 테이블을 원장 DB에 두고 플랫폼이 쓰기, 원장이 읽기. 스키마는 버전 관리되는 계약. | 체결 경로 네트워크 홉 제거, 신청 즉시 반영의 트랜잭션 보장. 직접 쓰기가 금지될 경우 EAI 전송으로 전환 가능하도록 스키마를 중립적으로 정의 |

## 3. 전체 아키텍처

```
┌─────────────────────────────────────────────┐
│  ① 수수료 룰 플랫폼 (신규, 원장 외부)              │
│                                             │
│  관리 웹 (현업용)          룰 스토어             │
│  - 이벤트/협수/기본 등록      - FeeRule          │
│  - Excel bulk 업로드       - FeeSchedule      │
│  - 승인(기안→결재)          - Enrollment       │
│                                             │
│  바인딩 엔진               조건 평가 배치         │
│  - 최저가 선택             - 협수 6개월 조건      │
│  - 즉시/배치 바인딩 생성      - 휴면복귀 감지       │
│  - 검증(reconciliation)   - 만료 처리          │
└──────────────┬──────────────────────────────┘
               │ 바인딩 테이블 쓰기 (계약)
┌──────────────▼──────────────────────────────┐
│  ② 수수료 바인딩 테이블 (원장 DB 내)              │
│     계좌 × 품목범위 → 요율표ID, 적용기간, 근거      │
└──────────────┬──────────────────────────────┘
               │ 단순 조회 (읽기 전용)
┌──────────────▼──────────────────────────────┐
│  ③ 원장 체결 모듈 (기존, 최소 수정)               │
│     체결 → 바인딩 조회 → 요율표 적용(구간 판정)     │
│     → 거래내역/잔고 반영 (near-instant)         │
└─────────────────────────────────────────────┘
```

원칙: **원장은 룰을 모른다.** 원장이 아는 계약은 "계좌+품목으로 바인딩을 조회하면 요율표가 나온다"뿐이다. 새 이벤트 유형은 전부 플랫폼(바인딩을 만들어내는 쪽)이 흡수한다.

## 4. 도메인 모델

### 4.1 FeeRule (수수료 룰)

기본/이벤트/협수의 공통 구조.

| 필드 | 내용 |
|---|---|
| 유형 | `BASE`(등급 기본) / `EVENT`(이벤트) / `NEGOTIATED`(협의) |
| 기간 | 시작일~종료일. 협수는 연장 시 기간 갱신 이력 보존 |
| 적용형태 | `신청형` / `가입형`(계좌·서비스 가입 시 자동) / `휴면복귀형`(복귀 감지 시 자동) / `일괄적용형`(대상 전원 즉시) |
| 적용범위 | ScopeSelector 참조 |
| 요율 | FeeSchedule 참조 |
| 상태 | 기안 → 승인대기 → 승인(활성) → 종료 (maker-checker) |

### 4.2 ScopeSelector (적용범위)

차원 계층: 상품군 → 거래소 → 세션 → 결제통화 → 품목(기초자산: 6A, 6B 등) → 종목.
각 차원은 `전체(*)` / `포함 리스트` / `제외 리스트`로 지정한다.
예: "CME 전체, 단 6E 제외" = 거래소:CME + 품목제외:[6E].
품목 수백 개 bulk 지정은 포함 리스트를 Excel 업로드로 채우는 것으로 해결한다.

### 4.3 FeeSchedule (요율표)

- **구성요소별 분해**: 자사 수수료분 / 유관기관분(거래소·예탁원·증금 등 기관별 라인) / 제세금.
- 각 구성요소에 **부담주체**: `고객부과` / `회사부담` / `면제`.
- 요율 방식: `정률(bp)` / `정액(계약당·주문당)` / `구간표`(국내옵션: 체결가 구간별 요율) / `정률+최소수수료`(해외주식).
- 통화·세션별 차등은 요율표 내부의 키 차원으로 표현한다.

### 4.4 Enrollment (대상 등록)

계좌 × 룰 연결. 생성 경로: 신청형=신청 이벤트, 일괄형=계좌 리스트 Excel bulk 업로드, 가입형/휴면복귀형=트리거 감지. 신청일·경로·상태를 보존한다.

### 4.5 NegotiatedCondition (협수 유지조건)

- 지표: `6개월 평균 자산`(해외주식) / `6개월 약정액`(그 외) — 롤링 윈도우.
- 임계값 + 평가주기 + 충족 시 액션(자동 연장 또는 담당자 승인 후 연장).
- 평가 결과 이력 보존 → 기존 엑셀 sum-up 수작업 대체.

### 4.6 FeeBinding (바인딩) — 원장과의 계약

`계좌 × 범위키 → 요율표ID, 유효기간, 근거 룰ID, 생성사유`.
원장이 조회하는 유일한 테이블. 최저가 경쟁의 **결과**만 담는다.

## 5. 바인딩 엔진

### 5.1 갱신 트리거

| 트리거 | 반영 시점 |
|---|---|
| 이벤트/협수 룰 승인·수정·조기종료 | 즉시 (해당 룰 대상 계좌 전체 재평가) |
| 고객 신청 (신청형) | 즉시 (해당 계좌만) |
| 계좌 등급 변경 | 즉시 |
| 신규 계좌 개설 (가입형) | 즉시 |
| 휴면복귀 감지 | 감지 시 (일 배치 또는 로그인 이벤트 연동) |
| 협수 조건 평가 결과 (연장/미충족 종료) | 평가 주기마다 |
| 룰 기간 만료 | 일 배치 (만료일 자정) |

### 5.2 최저가 선택

1. 해당 계좌가 대상인 활성 룰 수집
2. 범위가 겹치는 룰들을 범위 조각별로 분해
3. 조각마다 고객부과 총액 기준 비교 (지배관계 검증 덕분에 항상 판정 가능)
4. 승자를 바인딩으로 확정. 동률 tie-break: 협수 > 이벤트 > 기본

### 5.3 가드와 부가 기능

- **역마진 가드**: 승인 시점에 `회사부담 유관기관분 > 자사 수취분`이면 경고 표시(차단 아님 — 의도적 역마진 이벤트 허용, 승인자 인지 하에 결재). 운영 중 일 배치가 이벤트별 역마진 누적액을 집계해 대시보드 노출.
- **사전 시뮬레이션**: 승인 전 "대상 계좌 N개, 최근 3개월 거래 기준 수수료 감소분 X원"을 계산해 기안서에 자동 첨부.
- **검증 배치(reconciliation)**: 매일 새벽 룰 전체로부터 바인딩을 재유도하여 실제 바인딩 테이블과 비교, 차이 발견 시 알림. 트리거 누락의 안전망.

## 6. 관리 플랫폼 (현업용 웹)

1. **이벤트 등록 위저드**: ①기본정보(이름·기간·적용형태) → ②적용범위(상품군/거래소/세션/통화 트리 체크 + 품목 검색 + Excel 품목 리스트 업로드) → ③요율표(구성요소별 입력, 기존 요율표 복사 후 수정) → ④대상(전체/조건/계좌 리스트 Excel bulk 업로드) → ⑤시뮬레이션 결과 확인 → ⑥기안 상신
2. **승인함**: 결재자용. 시뮬레이션·역마진 경고·지배관계 검증 결과가 첨부된 상태로 승인/반려
3. **이벤트 현황 대시보드**: 활성 이벤트 목록, 대상자 수, 기간 진행률, 수수료 감면 누적액, 역마진 현황
4. **협수 관리**: 계좌별 협수 목록, 6개월 조건 충족 현황(자산/약정액 게이지), 만료 임박 알림, 조건 평가 이력, 원클릭 연장 상신
5. **계좌 조회**: 계좌 입력 시 "어떤 룰이 걸려 있고 품목별로 어떤 요율이 왜 적용되는지" 근거 포함 조회 (민원 대응용)
6. **감사 로그**: 모든 등록·수정·승인 이력

모든 리스트형 입력(품목, 계좌)은 **Excel 템플릿 다운로드 → 업로드 → 검증 미리보기 → 확정**의 동일 패턴을 사용한다.

## 7. 원장 연계 & 마이그레이션

- 바인딩 테이블은 원장 DB에 두고 플랫폼이 쓰기, 원장 모듈이 읽기. 스키마는 계약으로 버전 관리.
- 원장 체결 모듈 수정은 한 곳: "기존 등급 요율 조회" → "바인딩 조회(없으면 기존 등급 fallback)".
- 마이그레이션 순서:
  1. 기존 등급표를 BASE 룰로 이관
  2. **shadow-run**: 바인딩 결과와 기존 방식 수수료를 병행 계산·비교, 차이 0 확인
  3. 원장 조회 전환
  4. 신규 이벤트부터 플랫폼으로만 등록
  5. 기존 진행 중 이벤트·협수 이관
- 협수 조건 평가용 6개월 자산/약정액 지표는 원장 집계 테이블에서 일 단위로 플랫폼에 적재.

## 8. 검증 전략

- 요율 계산 로직: 상품군별 골든 테스트 케이스(실제 체결 사례 → 기대 수수료)
- 지배관계 검증·최저가 선택: 구간표 교차 케이스 포함 단위 테스트
- shadow-run 기간 일치율 100%가 전환 조건

## 9. 이번 산출물 범위 (v0)

관리 플랫폼 화면 1~5를 **mock 데이터 기반 프로토타입 웹앱**으로 구현하여, 전체 워크플로우(이벤트 등록 → 승인 → 바인딩 생성 → 계좌 조회로 적용 확인)를 클릭해볼 수 있게 한다. 용도: 사내 설득 및 요구사항 검증.

기술 스택 권고: 사내 표준이 있으면 그것을 따른다. 없으면 실 시스템은 Java/Spring Boot + React, 프로토타입은 단일 웹앱으로 빠르게 구성한다.
