# v0.5 — 규모 대응 수수료 아키텍처 (해석형 + 가변 feeKey) 설계

- 작성일: 2026-07-04
- 상태: **초안 — 사용자 검토 대기** (구현 착수 전)
- 전제: v0.4(branch feature/v0.2-instrument-master, `f353553`) 위의 근본 재설계
- 목적: 국내 1위 리테일 증권사 규모(**수천~수억 계좌 × 다수 거래소 × 세션 × 채널 × 품목**)에서 성립하는
  수수료(BASE/이벤트/협의수수료) 구조. 전량 materialize(계좌×품목)를 폐기하고 **룰로 저장 → 체결 시점 해석
  (resolve) → 진짜 예외만 계좌 단위 overlay**로 전환한다. 읽기 지연은 read-through 캐시로 유지.

## 배경: 왜 재설계인가

v0.4까지의 `rebindAll`은 **모든 계좌 × 모든 품목** 바인딩을 물리적으로 펼친다. 4계좌 데모엔 무해하나 실규모
(수억 계좌 × 수천 품목 = 10^10~10^11 행)에선 저장·재계산이 불가능하다. 개념(룰 경쟁 → 최저가 → 원장은
해석된 답만 조회)은 유지하되, **물리 설계를 해석형으로** 바꾼다. (근거: 성능은 캐시 적중 시 전량 조회와
동일, 저장/쓰기는 수 자릿수 절감 — §6.)

## 1. 해석 키 (feeKey) — 상품군별 가변 입도 (확정)

체결의 수수료를 결정하는 정규 키. **상품군마다 축이 다르다.**

| 상품군 | feeKey 축 | 예시 키 |
|---|---|---|
| 국내주식 | 거래소 · 세션 · 채널 | `KR주식\|KRX\|정규\|MTS` |
| 해외주식 | 시장(거래소) · 세션 · 채널 | `US주식\|NASDAQ\|주간\|MTS` |
| 국내파생 | 거래소 · 세션 · 채널 · **품목** | `KR파생\|KRX\|정규\|MTS\|KOSPI200선물` |
| 해외파생 | 거래소 · 세션 · 채널 · **품목** | `US파생\|CME\|야간\|HTS\|6A` |

- **주식은 품목을 붕괴**(키에서 제외) → 주식 feeKey 수 = 거래소×세션×채널 = 수십 개. 카디널리티가 가장 클
  뻔한 곳(주식=수천 종목×수억 계좌)에서 품목을 없애는 것이 규모 핵심.
- **파생만 품목 유지**(파생 유니버스는 주식보다 작음).
- **채널은 전 상품군 필수 축**(HTS/MTS/WTS/영업점/전화). 한국 리테일 수수료 최대 변수(온라인 vs 오프라인).

### 정규 차원 값(초안 — 검토 시 확정)
- 거래소/시장: 국내주식 {KRX, NXT}, 해외주식 {NASDAQ, NYSE, HK, TSE, SSE…}, 국내파생 {KRX}, 해외파생 {CME, EUREX, SGX, HKEX…}
- 세션: 국내주식 {정규, 시간외}, 해외주식 {정규(현지), 주간(주간거래)}, 국내파생 {정규, 야간}, 해외파생 {주간, 야간}
- 채널: {HTS, MTS, WTS, 영업점, 전화} (필요 시 {온라인, 오프라인}으로 축약 가능)

## 2. 요율표 구성요소 — 한국 리테일 표준 (상품군별)

기존 `FeeComponent`(kind 자사/유관기관/세금 × payer 고객부과/회사부담/면제 × rateType 정률/정액/구간표 +
minFee) 모델을 그대로 재사용. 이벤트·협의가 건드리는 것은 **위탁수수료(자사)** 뿐 — 유관기관·세금은
pass-through/면제불가.

- **국내주식**: 위탁수수료(자사, 정률 bp, *채널별*, 이벤트/협의 대상) · 유관기관제비용(유관기관, 정률,
  pass-through) · 증권거래세(세금, 매도, **거래소별 KRX/NXT 상이**, 면제 불가)
- **해외주식**: 위탁수수료(자사, 정률 + **minFee(USD)**, *시장별*) · 현지제비용/SEC(유관기관, 정률) ·
  (선택) 환전스프레드(자사, 정률)
- **국내파생**: 위탁수수료(자사, **계약당 정액** 또는 약정 정률, *품목별*) · 거래소수수료(유관기관, 계약당 정액)
- **해외파생**: 위탁수수료(자사, **계약당 정액(USD)**, *품목별*) · 거래소/청산수수료(유관기관, 계약당 정액)

`calcFee`(금액 계산)는 어느 방식이든 체결 시점에 동일하게 수행(요율표 × 체결). 재설계 범위 아님.

## 3. 3계층 룰 모델 (이벤트 구조 유지)

- **BASE (기본)**: feeKey scope별 요율표. 주식 = (상품군·거래소·세션·채널), 파생 = (+품목). 소수 테이블.
  계좌당 저장 없음. 기본값 fallback.
- **EVENT (일괄)**: `scope`(feeKey 축의 부분집합 — 거래소/세션/채널/품목) × **세그먼트 조건**(계좌 속성
  predicate) × 요율표 × 기간 × 우선순위. **룰로만 저장, 계좌 전개 안 함.** (v0.4의 이벤트 구조 유지, scope에
  채널 축 추가 + 상품군별 dimension-relevance로 품목 열림/닫힘.)
- **협의수수료 (NEGOTIATED)**: (계좌 × scope) overlay. **신청 + 조건** 게이트(최초 신청→조건확인→승인,
  연장 자동). 소수 — `nego_exception` 테이블에 계좌ID 인덱스로 저장. 전체 계좌 수와 무관.

### 이벤트/협의 예시
```
E1 신규계좌 국내주식 온라인 위탁 평생우대
   scope{국내주식, 채널:[HTS,MTS]} × 세그먼트{신규계좌} × 요율표{위탁 0.0140%}
E2 NXT 국내주식 수수료 인하
   scope{국내주식, 거래소:NXT} × 세그먼트{전체} × 요율표{위탁 인하}
E3 여름 CME 6A/6B 계약당 인하
   scope{해외파생, 거래소:CME, 품목:[6A,6B]} × 세그먼트{전체} × 요율표{계약당 1,500}   ← 품목 살아있음
N1 VIP 해외주식 협의수수료
   협의 overlay · scope{해외주식, 시장:US} · 조건{6개월평균자산 ≥ 5억} · 신청+승인
```

## 4. 저장 모델 (계좌 수와 무관하게 작음)

| 테이블 | 키 → 값 | 규모 |
|---|---|---|
| `base_schedule` | feeKey-scope(상품군·거래소·세션·채널[·품목]) → 요율표 | 수십~수백 |
| `event_rule` | rule_id → {세그먼트 조건, scope, 요율표, 기간, 우선순위} | 활성 이벤트 수 |
| `scope_index` | feeKey(또는 버킷) → [후보 event_rule_id] | event_rule에서 파생 |
| `nego_exception` | (계좌, scope) → 요율표, 기간 (**계좌ID 인덱스**) | 협의 체결 건수 |
| `resolved_cache` | (계좌, feeKey) → 요율표, 계산시각 | 실제 거래 working set |

## 5. 해석(읽기) 및 갱신(쓰기) 경로

**읽기(체결) — resolve(계좌, feeKey):**
1. `nego_exception[계좌]` 중 scope 매칭 → 후보 추가 (인덱스 1회, 대개 miss)
2. `scope_index[feeKey]` → 후보 이벤트(짧은 리스트) 중 계좌가 **세그먼트 조건** 만족하는 것만
3. `base_schedule[feeKey-scope]` (항상 존재)
4. 후보 중 **최저가** 선택 → `resolved_cache`에 기록
- hot 조합은 캐시 적중 = 단일 조회. cold/무효화 후만 위 해석(수십 µs).

**쓰기(배치/변경) — 전량 rebind 없음:**
- 룰 발효/만료/변경 → `event_rule`+`scope_index` 갱신, 그 scope의 캐시만 **무효화**(lazy: stale 표시).
- 계좌 지표 재산정(세그먼트 변동) → 계좌 속성만 갱신, 그 계좌 캐시만 무효화. **그리드 재작성 없음.**
- 협의 승인/해지 → `nego_exception[계좌]` 한 건 O(1).
- 배치 비용 = O(바뀐 룰) + O(무효화 캐시) — 변경분에 비례, 총량 아님.

## 6. 성능 근거

- 읽기 경로 = 인덱스 2~3회 + 후보 몇 개의 조건 비교 + 최저가 (룰 전체 스캔 아님). 인메모리 수 µs~수십 µs.
- 캐시 적중률 높음(계좌는 같은 품목 반복 거래) → 정상 상태 지연 ≈ 전량 pre-binding.
- 체결 경로엔 이미 DB 쓰기·리스크·네트워크(ms) 존재 → 수 µs 해석은 노이즈.
- 전량 pre-binding은 이 규모에서 "더 빠른 대안"이 아니라 **만들 수 없는 것.** → 성능 저하가 아니라 성능
  유지 + 규모 확보.
- 성립 조건: (a) 계좌 속성 사전계산(=지표 재산정 배치), (b) 짧은 후보(scope_index), (c) hot 캐시.
- 느려지는 경우와 회피: 조건을 체결 시 라이브 집계 → 금지(사전계산). 넓은 룰 변경 시 캐시 대량 무효화 →
  eager 삭제 대신 **lazy(다음 조회 때 재계산)**.

## 7. 프로토타입 → v0.5 갭

1. `ScopeSelector`에 **채널 축** 추가 + 상품군별 **dimension-relevance를 feeKey 입도에 적용**(파생만 품목,
   주식은 품목 붕괴). v0.2 매트릭스의 확장.
2. 바인딩/해석을 **feeKey 단위**로 재정의(주식 품목 붕괴).
3. `rebindAll`(전량) 폐기 → **resolve + read-through 캐시 + 증분 무효화** 도입. `explainBinding`도 resolve
   기반으로(수수료 결정 흐름 탭은 유지 — 후보/최저가/캐시 적중까지 시연).
4. mock을 **4상품군 표준 요율표 + 채널/세션 차원**으로 재구성(§2).
5. 배치 플로우(v0.4)를 **증분 무효화** 모델로 갱신(⑤가 전량 rebind 대신 무효화+재해석 시연).

## 8. 단계화(권장)

규모가 커서 서브프로젝트로 나눈다. 각 단계가 독립적으로 동작·검증 가능:
- **v0.5a 데이터 모델**: feeKey/ScopeSelector(채널·가변 입도) + 4상품군 표준 요율표 mock + dimension-relevance.
- **v0.5b 해석 엔진**: resolve(nego→event→base 최저가) + scope_index + read-through 캐시 + 증분 무효화.
  `explainBinding`/계좌 조회/배치 플로우를 resolve 기반으로 이관.
- **v0.5c 화면 반영**: 위저드 scope에 채널, 배치 플로우를 무효화 모델로, 캐시 적중/후보 리스트 시연.

## 9. 범위 외

- 실제 원장/체결 로그 연동, 분산 캐시(Redis 등) 실물, 트랜잭션/동시성, 인증/권한.
- FX 실시간 환율, 세금 실제 세율 테이블(고정 mock 유지).

## 10. 열린 결정 (검토 시 확정)

1. 정규 차원 값(§1) — 세션·거래소·채널 목록 확정(특히 국내주식 세션을 정규/시간외/NXT까지 어디까지).
2. 채널 입도 — 세분(HTS/MTS/WTS/영업점/전화) vs 축약(온라인/오프라인).
3. 단계화(§8) 범위 — v0.5a부터 순차 진행할지, a+b를 한 번에.
4. resolved_cache를 프로토타입에서 실제 구현(무효화 시연)할지 vs 개념만 문서화.
