# 수수료정책 과제 ISMP 산출물 (참조 표본) — v0.2

- 과제: 차세대 원장 수수료 체계 재설계 (fee 도메인)
- 문서 성격: ISMP(정보화전략계획) 산출물 4종 통합본. **fee 과제를 참조 표본으로 먼저 완성**하고, 승인된 깊이를 나머지 과제 템플릿으로 복제하기 위한 기준 문서입니다.
- 근거 소스: 본 저장소 HEAD `4230e81` (문서 라인 v1.5) 기준으로 갱신했습니다. 사실 근거는 `src/domain/*`·`src/masterdata/*`(확정 로직)와 `docs/superpowers/specs/*`(v0.5~v0.9 설계 스펙)입니다.
- 작성 기준일: 2026-07-08

## 0. 문서 개요

### 0.1 상세화 천장 (반드시 준수)

- 본 문서의 상세화 수준은 **ISMP 수준**입니다. 즉 "발주사·벤더가 입찰·검수하는 데 필요한 선"까지만 기재합니다.
- **그대로 코딩 가능한 수준(테이블 컬럼, 화면 레이아웃, 산출 알고리즘, 시퀀스, API 스펙, 물리 배치도, 사이징 수치)은 기재하지 않습니다.** 이는 구축 단계(SI) 귀속 산출물입니다.
- 저장소의 상세설계서(기술설계서 v1.5, 테이블 설계서, 업무 프로세스 정의서)는 근거·판정용 참조 자료이며, 그 깊이를 본 산출물에 그대로 옮기지 않습니다.
- **자가 점검 1문**: "이 문장이 벤더 입찰·검수에 필요한가, 아니면 벤더가 *어떻게* 만들지를 당사가 정해주는가?" — 후자면 삭제하고 구축 단계로 이관합니다.

### 0.3 표기 규칙

- 개조식·합쇼체로 기술하며, 주어는 당사(발주사) 또는 시스템입니다.
- **【확인 필요】**: 코드·설계 문서에 근거가 없어 당사·현업·컨설팅이 확정해야 하는 항목입니다. **날조하지 않고 비워 둡니다.** 미결 항목 총괄은 부록 A에 있습니다.

## 2. 산출물 ② 기능분해

기능을 계층으로 나열하고 소기능마다 1~3행 설명과 요구 ID를 참조합니다. 처리 로직·시퀀스·API 스펙은 기재하지 않습니다.

### 2.1 정책 관리
- **2.1.1 이벤트 등록** — 위저드로 기본정보·적용범위·요율표·대상·시뮬레이션을 입력해 정책을 기안합니다. (FR-POL-01, FR-POL-03, FR-POL-04)
- **2.1.2 정책 승인/반려** — 결재자가 검증 결과(지배관계·역마진·시뮬레이션)를 확인해 승인 또는 반려합니다. (FR-POL-02, FR-DOM-01, FR-DOM-02)
- **2.1.3 정책 수명주기 전이** — 기간 도래·만료에 따라 활성/종료 상태를 전이합니다. (FR-POL-02)
- **2.1.4 요율표 구성** — 구성요소(자사·유관기관·세금)와 방식(정률·정액·구간표)·부담주체를 정의합니다. (FR-POL-01)

### 2.2 협의수수료
- **2.2.1 협의 신청** — 계좌 단위로 적용범위·요율표·대상 계좌를 입력해 협의를 신청합니다. (FR-NEG-01)
- **2.2.2 자격 자동판정** — 상품군 표준 기준으로 계좌별 자격 충족을 판정하고 미충족은 영업 예외로 표시합니다. (FR-NEG-01, FR-COND-01)
- **2.2.3 협의 승인/반려** — 요청 묶음 단위로 승인/반려하고 승인 시 활성화·캐시 무효화합니다. (FR-NEG-01, FR-CACHE-02)
- **2.2.4 협수 조건 평가·연장** — 6개월 지표로 유지/탈락을 분류하고 연장 여부를 결정·기록합니다. (FR-COND-01)

### 2.3 요율 해석 엔진
- **2.3.1 수수료키 구성** — 상품군별 입도로 체결 컨텍스트에서 수수료키를 조립합니다. (FR-KEY-01)
- **2.3.2 요율 해석(resolve)** — 협의 우선→우선순위→기본 순으로 자격 통과 최저가 정책을 선택합니다. (FR-RES-01, FR-RES-02)
- **2.3.3 정책 우선순위 산정** — 계좌 무관 정책의 요율 순위를 사전 산정·메모이즈합니다. (FR-RANK-01, FR-RANK-02)
- **2.3.4 수수료 금액 산출** — 선택된 요율표를 체결가·수량에 적용해 고객부과/회사부담 금액을 계산합니다. (FR-LEDGER-01)

### 2.4 캐시·무효화
- **2.4.1 해석 결과 캐시** — 계좌×수수료키 해석 결과를 read-through로 적재·조회합니다. (FR-CACHE-01)
- **2.4.2 범위/계좌 무효화** — 정책 변경은 범위 단위, 지표 변동은 계좌 단위로 캐시를 무효화합니다. (FR-CACHE-02)
- **2.4.3 우선순위 인덱스 무효화** — 정책·요율표 변경 시에만 인덱스를 무효화합니다. (FR-CACHE-03)

### 2.5 검증
- **2.5.1 지배관계 검증** — 후보 요율표가 기준 요율표를 전 가격대에서 지배하는지 검사합니다. (FR-DOM-01)
- **2.5.2 역마진 판정** — 회사부담이 자사 수취를 초과하는지 판정·경고합니다. (FR-DOM-02)
- **2.5.3 사전 시뮬레이션** — 대상 계좌 수·예상 감면액을 상신 전 산출합니다. (FR-DOM-01)

### 2.6 마스터·원장 연계
- **2.6.1 종목/상품 마스터** — 종목 속성·상장 상태를 관리하고 상품(거래 가능 단위)을 파생합니다. (FR-MST-01)
- **2.6.2 원장 동기화** — 신규 상장 종목을 결정적으로 유입합니다. (FR-MST-01)
- **2.6.3 체결 시 요율 적용** — 원장이 요율 조회 계약으로 적용 요율을 얻고 없으면 폴백합니다. (FR-LEDGER-01)

### 2.7 이력·감사
- **2.7.1 근거 추적** — 적용 요율의 출처 정책을 제시합니다. (FR-TRACE-01)
- **2.7.2 변경 이력** — 정책 상태 전이·바인딩 변경·조건 평가를 이력화합니다. (FR-TRACE-01, NFR-AUDIT-01)
- **2.7.3 정합성 점검(reconciliation)** — 정본으로부터 재유도한 결과와 실제를 비교해 불일치를 통지합니다. (NFR-CONSIST-01)

## 3. 산출물 ③ 아키텍처 방향

핵심 결정마다 근거·대안·리스크를 기술합니다. 물리 배치도·사이징·샤드 수·구현 알고리즘은 구축 단계 귀속이므로 제외합니다.

### 결정 A. 요율 산출 = resolve-on-read + 범위 기반 캐시 무효화
- **결정**: 계좌×상품 전량 사전 전개(pre-binding materialize)를 폐기하고, 정책은 정책으로만 저장하며 체결 시점에 해석(resolve)합니다. 결과는 read-through 캐시로 적재하고, 정책/지표 변경 시 영향 범위(scope)·계좌 단위로만 캐시를 무효화합니다.
- **근거**: 전량 전개 시 계좌×품목 조합이 곱셈적으로 폭발합니다(v0.5 스펙: 실규모에서 수억 계좌 × 수천 품목 규모의 행 = 저장·재계산 불가). 조회형 경로는 후보 소수 비교로 끝나 규모와 무관하게 준실시간을 유지합니다.
- **대안**: 전량 pre-binding. → v0.5 스펙 결론상 이 규모에서 "더 빠른 대안"이 아니라 "만들 수 없는 것"이므로 기각. 캐시 적중 시 지연은 pre-binding과 대등합니다.
- **리스크**: 넓은 정책 변경 시 무효화 대상 스캔 비용이 커질 수 있습니다. 무효화를 즉시 삭제(eager)로 할지 다음 조회 시 재계산(lazy)으로 할지는 실규모 검토가 필요합니다(프로토타입은 eager, 설계 스펙은 lazy 제안). → 구축 시 확정.

### 결정 B. 수수료키 입도의 상품군별 비대칭
- **결정**: 수수료키 입도를 상품군별로 다르게 둡니다. 주식군은 품목 차원을 붕괴(거래소·세션·채널)하고, 파생군은 품목 차원을 유지합니다.
- **근거**: 카디널리티가 가장 큰 지점(주식=수천 종목 × 대량 계좌)에서 품목을 제거하는 것이 규모 억제의 핵심입니다(v0.5 스펙). 증권유형·세제·개별종목을 획일적으로 결합하던 안티패턴을 제거합니다.
- **대안**: 전 상품군 획일 키(품목 포함). → 주식에서 조합 폭발을 유발하므로 기각.
- **리스크**: 상품군별 세션 등 차원의 유효값 매핑이 확정되어야 합니다(v0.5 스펙: "필요 시 상품군별 유효값 매핑"으로 잔여). → 구축 시 확정.

### 결정 C. 요율 캐시 2계층 (우선순위 인덱스 + 계좌축 해석 캐시)
- **결정**: 캐시를 2계층으로 둡니다. (1) 계좌 무관 정책 우선순위 인덱스(전역, 정책·요율표 변경 시 재빌드), (2) 계좌×수수료키 해석 결과 캐시(계좌축).
- **근거**: 우선순위 인덱스는 `(정책, 요율표)`의 순수 파생값이라 계좌·체결가와 무관하게 재사용이 안전합니다(v0.8·v0.9 스펙). 값비싼 축(계좌×수수료키)만 별도 캐시로 흡수합니다.
- **대안**: 매 조회 시 전체 정책 재정렬. → 캐시 미스마다 O(N log N) 반복으로 낭비(v0.9 스펙 갭 분석)이므로 기각.
- **리스크**: 다중 인스턴스 수평 확장 시 로컬 캐시 정합성(분산 무효화)이 별도 과제입니다(v0.9 스펙 범위 밖). → 부록 A-06.

### 결정 D. 캐시/정본 경계
- **결정**: 파생·재생성 가능한 투영만 캐시하고, authoritative 요율 정본은 내구 저장소에 둡니다. 캐시가 단독 정본이 되지 않도록 합니다.
- **근거**: 판단 기준은 "durable state의 순수 함수인가"입니다. 순수 함수로 재유도 가능한 것(해석 결과·우선순위 인덱스)만 캐시하며, 이는 정합성 안전망(재유도 후 비교)의 전제가 됩니다.
- **대안**: 우선순위 인덱스를 전용 영속 테이블로 둠. → 이중 진실원 회피·"원장은 룰을 모른다" 원칙상 기본은 파생 캐시. 단 다중 인스턴스·시점 감사·화면 직접 급여가 필요하면 전용 테이블이 정당화됩니다(v0.9 스펙 예외 조건).
- **리스크**: 정본/캐시 경계의 물리적 배치(내구 저장소 선정·계약 테이블 소유권)는 원장 기술 스택 확인 후 확정됩니다. → 구축 시 확정.

### 결정 E. 정산 시 플랫폼 미가용 거동 (열화 모드)
- **결정**: **미정입니다.** → **【확인 필요】**
- **근거**: 코드상 체결 조회 실패 시 기존 등급 요율로 폴백하는 안전망(FR-LEDGER-01)은 있으나, 정산 시점 플랫폼 미가용 시의 열화 모드 정책(중단/지연/폴백 정산/사후 정정)은 저장소에 근거가 없습니다.
- **대안 / 리스크**: 열화 모드 정책이 확정되어야 가용성 목표(NFR-AVAIL-01)와 폴백 회계 처리 범위가 정해집니다. → 부록 A-05, 당사·현업·원장팀 협의 필요.

### 결정 F. 매체(채널)는 바인딩 키, 세션은 요율표 내부 차원
- **결정**: 매체(주문채널)는 요율 적용의 키 차원으로, 세션은 요율표 내부 차원으로 취급합니다.
- **근거**: 매체 한정 이벤트는 매체에 따라 최저가 승자 정책 자체가 달라지므로 키로 분리합니다. 세션 차등은 같은 승자 정책 안의 요율 차이이므로 요율표 내부로 흡수합니다(`table-design.md` 4.5/4.6, 결정 근거만 인용).
- **대안**: 세션도 키 차원으로. → 승자가 세션에 따라 바뀌지 않으므로 불필요한 키 폭발이라 기각.
- **리스크**: 두 차원의 취급 차이가 원장 조회 계약에 반영되어야 하며, 물리 키 전개는 구축 귀속입니다. → 구축 시 확정.

## 4. 산출물 ④ RFP 반영

수수료 체계 재설계 사업의 발주 범위·제약·평가 기준을 정의합니다. **확정 설계를 못박아 벤더의 가치 부가 여지를 제거하지 않습니다** — 성과 목표와 제약을 정의하고 물리 설계·사이징·구현은 벤더 과업으로 남깁니다.

### 4.1 사업 범위 (벤더 구축 대상)
- 3계층 정책 모델(BASE/EVENT/NEGOTIATED)과 상태 수명주기의 구축. (요구 참조: FR-POL-01~04)
- 상품군별 비대칭 수수료키에 기반한 요율 해석 엔진과 캐시·무효화 체계 구축. (FR-KEY-01, FR-RES-01~02, FR-RANK-01~02, FR-CACHE-01~03)
- 협의수수료 신청·자격 판정·승인·조건 평가/연장 업무 기능 구축. (FR-NEG-01, FR-COND-01)
- 등록 시점 지배관계·역마진 검증 및 사전 시뮬레이션 구축. (FR-DOM-01~02)
- 종목/상품 마스터·원장 동기화 및 체결 시 요율 적용 계약 연계. (FR-MST-01, FR-LEDGER-01)
- 근거 추적·변경 이력·정합성 점검(reconciliation) 구축. (FR-TRACE-01, NFR-AUDIT-01, NFR-CONSIST-01)

### 4.2 요구 참조
- 기능 요구: 1.1장 FR 전체.
- 비기능 요구(원칙): 1.2장 NFR 전체. 정량 목표는 4.4·부록 A로 벤더 제안·협의 시 확정.

### 4.3 제약 사항 (벤더 준수)
- **원장 비대칭 소유**: 요율 조회 계약 테이블은 쓰기=플랫폼, 읽기=원장 전용이며 하위호환을 원칙으로 함. 원장은 정책 평가 로직을 알지 않음. (NFR-COMPAT-01)
- **원장 결합 경로 중립**: 직접 DB 쓰기/EAI 메시지 등 특정 연계 방식에 종속되지 않는 계약 구조를 유지. (`table-design.md` 4.4)
- **DBMS·물리 스택 미확정**: 원장 실제 기술 스택 확인 후 물리 설계 확정(현행 문서의 Oracle 표기는 기본값 가정). → 부록 A-07.
- **세션/채널 차원 규칙**: 매체는 키, 세션은 요율표 내부(결정 F)라는 원칙을 준수.
- **지배관계 전제**: 우대 정책은 기준 요율을 전 가격대에서 지배해야 하며, 교차 구간(가격대별 우열 역전) 동시 지원은 확정 대상. → 부록 A-04.

### 4.4 평가 기준 (제안 심사 관점)
- 규모 대응 설계 역량: 계좌×품목 전량 전개를 회피하는 해석형 코어 설계·구현 경험.
- 캐시·무효화 전략의 타당성: 범위/계좌 단위 무효화, 다중 인스턴스 정합성 대응 방안.
- 정합성·안전망: reconciliation·폴백 모니터링·감사 이력 설계.
- 원장 연계 안전성: 계약 하위호환·shadow-run/dual-write 등 무중단 전환 절차.
- 성능·가용성 목표 달성 방안: 정량 목표(부록 A) 충족 근거.

### 4.5 규모 근거 (제안 규모 산정용)
- 정책 저장 규모는 계좌 총량과 무관(기본 요율표 수십~수백, 활성 이벤트 수, 협의 건수). (v0.5 스펙)
- 값비싼 축은 계좌×수수료키(해석 캐시)이며, 주식 수수료키는 품목 붕괴로 상수 규모, 파생은 품목 유지. (FR-KEY-01)
- **정량 규모(계좌 수, 품목 수, 체결 TPS, 피크 배수, 데이터 보존기간)** → **【확인 필요】** 부록 A-01.
