# 부록 ⑤ 실현 가능성 증빙 (참조 구현)

- 목적: 본 ISMP가 요구하는 규모·성능·아키텍처 수준이 **실현 가능함**을 참조 구현과 실측으로 제시합니다.
- 성격: 이 부록은 설계를 확정하는 문서가 아닙니다. 참조 구현은 실현 가능성 증빙일 뿐이며, 물리 설계·사이징·구현 방식은 벤더 과업입니다.
- 기준: 브랜치 `feature/v1-java-postgres`. 동작하는 백엔드(`server/`)·프론트(`src/`)·테스트·성능 실측을 포함합니다.

## 1. 참조 구현이 증빙하는 것

| 요구 | 증빙 사실 | 근거 |
|---|---|---|
| FR-BIND-01·03 (배정판 사전 산출·3층) | 전체 재산출·일 delta·수시 증분 3경로가 **동일 승자 확정 코어를 공유**하여 동작 | `server/…/batch/`(BindingRebuilder·DeltaBatch·IncrementalBinder), 배치 실행 이력 |
| FR-BIND-02·F (우대분만 저장) | 기본 승자·미스 셀은 배정에 기록하지 않고, 미스 시 원장이 기본을 직접 해석 | `server/…/service/LedgerLookupService`(미스=기본 승자 결정, fallback 표기) |
| FR-RANK-01~03 (통합 랭킹·자격 게이트) | 기본·이벤트·협의를 한 순위로 정렬하고 계좌별 자격 게이트로 승자 확정(협의 무조건 우선 아님) | `server/…/domain/`(PolicyRanking·WinnerResolver·EligibilityGate) |
| FR-KEY-01~02 (비대칭 조회키·조회구분) | 파생은 품목까지, 주식형은 조회구분 단위로 셀 전개 | `server/…/domain/`(LookupKey·ScopeMatcher), 셀 유니버스 전개 |
| FR-DOM-01~02 (검증) | 등록 시점 지배관계·역마진 검증이 동작 | `server/…/domain/DominanceValidator`, 룰 검증 경로 |
| FR-TRACE-01 (결정 추적) | 후보·자격 게이트·탈락 사유·배정판 히트를 재구성 | `server/…/service/TraceService` |
| NFR-CONSIST-01 (정합성) | 전체 재산출이 기대 대 실제를 대조(insert/update/unchanged/delete 집계) | 배치 실행 결과 집계 |

※ 위 표는 "구현이 존재함"을 증빙으로 인용할 뿐, 각 클래스의 시그니처·컬럼·알고리즘을 ISMP 산출물로 채택하지 않습니다.

## 2. 성능 실측 (로컬 벤치마크)

- 방법: 계좌ID 선두 복합키·커버링 인덱스 등가 구조에서 무작위 계좌 1만 건 배정판 점조회. (설계서 v1.7 부록)
- **점조회 응답(중앙값 / 상위 1% / 최대)**

| 배정판 규모 | 중앙값 | 상위 1% | 최대 |
|---|---|---|---|
| 1,000만 행 | 0.041ms | 0.260ms | 1.2ms |
| 5,000만 행 | 0.202ms | 0.698ms | 2.5ms |
| 1억 행 | 0.269ms | 0.613ms | 2.5ms |
| 2억 행 | 0.293ms | 0.736ms | 10.2ms |

- **배치 적재**: 2억 행 정렬 순서 적재 = 단일 스레드 약 6분 실측. 무작위 순서 갱신은 인덱스 캐시 미스로 수십 배 느려짐 → "키 정렬 순서 적재"가 필수라는 설계 근거.
- **시사점**: 배정판 점조회는 인덱스 구조(B-tree 깊이 3~4단, 10억 행에서도 4~5단)상 행 수와 거의 무관하게 준실시간을 유지합니다. 이는 "배정판은 우대분만 저장 + 키 정렬 적재" 결정(결정 A·E·F)의 실현 가능성을 뒷받침합니다.

## 3. 실측의 한계 (실환경 목표치 아님)

- 벤치마크는 **로컬 단일 노드(SQLite 등가 구조)** 측정입니다. 원 벤치마크 스크립트는 저장소에 포함되어 있지 않으며, 수치는 측정 당시 기록으로만 보존됩니다.
- 실환경 목표치(피크 TPS, 동시성, 네트워크, 이중화, 운영 DBMS)는 다르며, **10억 행 이상은 실측이 아니라 B-tree 깊이 기반 추정**입니다.
- 따라서 실환경 성능·규모 목표는 벤더 제안·PoC로 확정해야 합니다. → `00_개요_및_산출물_인덱스.md` 부록 A-01·A-12(【확인 필요】).

## 4. 참조 구현 기동 개요 (감사·검증용)

- 구성: Spring Boot 3(백엔드 `server/`) + PostgreSQL 16 + React 프론트(`src/`). 스키마·시드는 마이그레이션으로 자동 적용.
- 목적: 산출물 검수 시 "요구가 실제로 동작함"을 확인하기 위한 참조. 운영 배포 인프라·인증·실 원장 연계는 범위 밖(A-08·A-11).
